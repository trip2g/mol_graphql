# $mol + GraphQL codegen: фрагменты в духе Relay, без React

Стартер для копирования: как связать компоненты **$mol** с GraphQL API со
**сквозной типизацией** и **фрагментами в стиле Relay** - объявленными в каждом
компоненте, подключаемыми по имени, замаскированными для всех остальных, без
единого импорта и без изменений в сборщике $mol/mam.

**Живое демо:** https://trip2g.github.io/mol_graphql/ (работает целиком в браузере:
входная точка `$demo_app_static` собирает приложение вместе с браузерным GraphQL-моком,
без сервера; лайки сбрасываются при перезагрузке).

```sh
docker-compose up --build
# open http://localhost:8080  (mock GraphQL API is on http://localhost:4000/graphql)
```

Вы получаете страницу, которая приветствует пользователя (простой типизированный
запрос), показывает список заметок (родительский запрос, компонующий фрагмент потомка)
и даёт лайкнуть заметку (типизированная мутация + реактивный перезапрос). Всё
типизировано от схемы до кода компонента.

## Идея в одном абзаце

Каждый модуль компонента держит свои GraphQL-операции рядом с кодом - отдельными
файлами `.graphql`. Codegen в watch-режиме (`graphql-codegen` + ~150 строк
собственного preset/plugin) генерирует рядом с каждым файл `<name>.graphql.ts`:
тонкую, полностью типизированную обёртку в глобальном `namespace $`. Сборщик $mol
(`mam`) компилирует эти сгенерированные файлы как обычный модульный TypeScript.
**Два сборщика встречаются через файловый шов**: graphql-codegen пишет
`.graphql.ts`, mam его компилирует - ровно так же, как в самом $mol работает
`.view.tree` → `-view.tree/*.d.ts`. Сгенерированный символ `$<module>_<opname>`
появляется в `namespace $` без единого импорта.

Этот репозиторий - пакет `demo` воркспейса [mam](https://github.com/hyoo-ru/mam)
(корень репозитория монтируется в `demo/`), поэтому путь `app/...` в репозитории -
это путь `demo/app/...` в воркспейсе, а сгенерированные символы носят префикс `$demo_`.

| исходный `.graphql` | сгенерированный `.graphql.ts` | экспортируемый символ |
|---|---|---|
| `app/notes.graphql` | `app/notes.graphql.ts` | `$demo_app_notes(): demo_app_notesQuery` |
| `app/viewer.graphql` | `app/viewer.graphql.ts` | `$demo_app_viewer(): demo_app_viewerQuery` |
| `note/card/note.graphql` | `note/card/note.graphql.ts` | `$demo_note_card_note` + `$demo_note_card_note_unmask(ref)` |
| `note/card/like.graphql` | `note/card/like.graphql.ts` | `$demo_note_card_like(vars): demo_note_card_likeMutation` |

Типы результата и переменных **вшивает генератор** (он знает схему и операцию),
поэтому нет зависимости от побайтового совпадения строковых литералов.

### Операции именуются автоматически по расположению файла

Можно написать `query { ... }` (анонимно) или `query AnyName { ... }`: codegen
переписывает имя операции на каноническое, выведенное из пути файла, - это
сгенерированный символ без ведущего `$`. Для `note/card/like.graphql` символ -
`$demo_note_card_like`, поэтому реально отправляется документ
`mutation demo_note_card_like(...)`. Вызываемая функция, имя операции, которое
видит сервер, и путь файла совпадают 1:1.

Отсюда бесплатно следуют две вещи: логи сервера и APM всегда получают осмысленное
имя операции, даже если разработчик забыл его написать; и привычная ошибка
graphql-codegen "anonymous operation" исчезает (в этом репозитории
[`app/viewer.graphql`](app/viewer.graphql) анонимен намеренно). Фрагменты НЕ
переименовываются: их подключают по имени из других файлов (модель Relay), и
переписывание сломало бы места подключения. Вместо этого codegen печатает
неблокирующее предупреждение, когда имя фрагмента отклоняется от того же
канонического, выведенного из пути; фрагменты этого репозитория названы
канонически (`note/card/note.graphql` объявляет `demo_note_card_note`).

## Фрагменты: модель Relay, стиль $mol

Здесь воспроизведена [модель рендеринга фрагментов Relay](https://relay.dev/docs/guided-tour/rendering/fragments/) -
без React и без его нормализованного стора:

- **Компонент объявляет свои потребности в данных именованным фрагментом** в
  собственном файле `.graphql`:

  ```graphql
  # note/card/note.graphql
  fragment demo_note_card_note on Note {
    id
    title
    body
    likes
    author { name }
  }
  ```

- **Фрагменты глобальны и подключаются по уникальному имени.** Дерево компонентов
  им безразлично: любая операция (или другой фрагмент) может подключить
  `...demo_note_card_note`. Имена фрагментов следуют тому же каноническому правилу
  из пути файла, что и операции: путь сам кодирует компонент и свойство
  (`note/card/note.graphql` -> `demo_note_card_note`), так что глобальная
  уникальность получается из пути автоматически. В отличие от операций, codegen
  только ПРЕДУПРЕЖДАЕТ (не отвергает и не переписывает), если имя фрагмента
  отклоняется: подключение в GraphQL идёт по имени и остаётся корректным при
  любом имени. На этапе кодогенерации
  определения фрагментов (транзитивно) вливаются в каждую операцию, которая их
  подключает, - получается один сетевой запрос без рантайм-реестра документов:

  ```graphql
  # app/notes.graphql
  query DemoAppNotes {
    notes {
      id
      ...demo_note_card_note
    }
  }
  ```

- **Маскировка.** Родитель физически получает данные фрагмента, но его ТИП их
  скрывает. `$demo_app_notes().notes[0]` типизирован как
  `{ id: string } & { ' $fragmentRefs'?: { demo_note_card_noteFragment } }`. Чтение
  `.title` в родителе - ошибка компиляции:

  ```
  error TS2339: Property 'title' does not exist on type
  '{ __typename?: "Note"; id: string; } & { ' $fragmentRefs'?: ... }'
  ```

- **Аксессор unmask вместо `useFragment`.** `useFragment` в Relay - на самом деле
  identity-каст, а не хук, поэтому в $mol это обычная сгенерированная функция,
  вызываемая внутри реактивного свойства `$mol_mem`:

  ```ts
  // note/card/card.view.ts
  export class $demo_note_card extends $.$demo_note_card {

      // opaque ref bound by the parent in view.tree: `note_ref <= card_ref*`
      note_ref(): $demo_graphql_ref<$demo_note_card_note> {
          return super.note_ref()
      }

      @ $mol_mem
      note() {
          return $demo_note_card_note_unmask(this.note_ref()) // typed fields, only here
      }

      note_title() { return this.note().title }
  }
  ```

### Соглашение: мутация перезапрашивает все запросы на странице

Ни нормализованного стора, ни машинерии консистентности кэша. Инвалидация - один
реактивный маркер ([`graphql/index.ts`](graphql/index.ts)): каждый запрос подписан
на счётчик поколений, каждая мутация его увеличивает, и все мемоизированные через
`$mol_mem` запросы текущей страницы перезапускаются. Это вся история про кэш, а
эргономика композиции и маскировки Relay остаётся с вами и без него.

Да, это порождает дублирующие и пересекающиеся запросы. Это осознанный размен:
лишний запрос дешевле, чем UX-баг из-за забытого коллбэка инвалидации -
классических граблей нормализованного кэша. По умолчанию дубликатов не бойтесь.
Оптимизируйте, только когда проявится реальная цена.

Когда придёт время оптимизировать: передайте `{ revalidate: false }` в
`$demo_graphql_request`, чтобы вывести один запрос из-под перезапроса или одну
мутацию из-под его запуска. На бэкенде persisted queries делают повторные чтения
дешёвыми. Слой запросов и путь unmask остаются маленькими и заменяемыми, так что
более умную пофрагментную реактивность через `$mol_mem` можно добавить позже,
не трогая сгенерированный код.

Чтобы увидеть это вживую: каждая карточка заметки показывает счётчик `renders`,
а рядом есть статическая панель с `revalidate:false`. Лайкните любую заметку -
счётчики всех карточек тикают вместе, а статическая панель стоит на месте
([`card.view.ts`](note/card/card.view.ts#L56-L69),
[`app.view.ts`](app/app.view.ts#L30-L60)).

### Выбор области перезапроса: режим codegen `revalidation`

Сколько перезапрашивает мутация - переключатель времени компиляции. Одна строка в
конфиге плагина ([`codegen/graphqlgen.js`](codegen/graphqlgen.js)) выбирает
метаданные инвалидации, которые вшиваются в каждую сгенерированную обёртку:

```js
config: { molRuntime: '$demo_graphql', revalidation: 'all' } // 'all' | 'by_typenames' | 'disable'
```

- **`all`** (значение по умолчанию, с ним и поставляется это демо): соглашение
  выше. Каждый запрос подписан на один универсальный маркер, каждая мутация его
  увеличивает.
- **`by_typenames`**: codegen проходит каждую операцию по схеме и вшивает в обёртку
  получившийся набор объектных типов. Запрос подписывается на маркер каждого типа,
  который читает (`reads: ['Note', 'User']`); мутация увеличивает маркеры типов из
  своего ответа (`writes: ['Note']`). Лайк (пишет `Note`) перезапрашивает список
  заметок (читает `Note`), но не запрос о пользователе (читает только `User`) - и
  для этой изоляции не нужен ручной `{ revalidate: false }`. А поскольку набор
  чтения берётся из обхода схемы, а не из данных рантайма, запрос, чей список
  пришёл пустым, всё равно перезапросится, когда появится первый элемент
  (document cache в urql, выводящий те же наборы из рантаймовых `__typename`,
  этот случай упускает).
- **`disable`**: обёртки нигде не подписываются и ничего не увеличивают. Запросы
  выполняются один раз и пересчитываются только при изменении собственных входов.

`{ revalidate: false }` работает в любом режиме и выводит из-под соглашения
отдельный вызов. Операции без метаданных (рукописные вызовы
`$demo_graphql_request`) в любом режиме деградируют до универсального маркера,
так что смешивать их со сгенерированными обёртками безопасно.

`by_typenames` можно включать, если ваш бэкенд гарантирует: каждая мутация
возвращает в своём payload все объектные типы, которые она затрагивает. Тогда
вычисленный набор записи (типы в ответе мутации) совпадает с реально затронутым
набором, пер-типовая инвалидация точна, и `@touches` не нужен. Мутация, в payload
которой нет ни одного объектного типа (delete, возвращающий `Boolean`),
автоматически откатывается к универсальному маркеру: эффект неизвестен - считаем,
что затронуто всё. Опасный случай - мутация, которая возвращает один тип, но
меняет и другой: например, лайк, который заодно увеличивает счётчик пользователя.
Либо включите изменяемый тип в payload, либо объявите его:

```graphql
mutation DemoFixtureTouch($id: ID!) @touches(types: ["User"]) {
  note_like(id: $id) { id likes }
}
```

Codegen объединяет объявленные типы с `writes` и вырезает директиву из
отправляемого документа. Честная оговорка: ничто не заставляет автора мутации с
побочным эффектом вспомнить про это объявление, а забытый `@touches` - это молча
устаревшие данные. Этот сценарий отказа - ровно тот «забытый коллбэк
инвалидации», который соглашение этой страницы и должно было сделать невозможным;
поэтому безопасным умолчанием остаётся `'all'`.

Два других режима доказаны тестами, а не выложенной страницей:
[`graphql/index.test.ts`](graphql/index.test.ts) гоняет обёртки, сгенерированные
в режимах `by_typenames` и `disable` ([`graphql/fixture/`](graphql/fixture/)),
на мок-транспорте и проверяет выборочный перезапрос, случай пустого списка,
`@touches` и деградацию без метаданных.

## Куда смотреть (маршрут чтения)

Пройдите по порядку, чтобы увидеть всю идею - от файла `.graphql` до работающего
компонента:

1. Собственные операции компонента: [`app/notes.graphql`](app/notes.graphql) и
   [`note/card/note.graphql`](note/card/note.graphql). Обычные файлы рядом с компонентом.
2. Codegen, который их типизирует: [`codegen/molplugin.js`](codegen/molplugin.js):
   [`renameOperations`](codegen/molplugin.js#L103-L115) переписывает имя операции на
   каноническое из пути файла;
   [`operationCode`](codegen/molplugin.js#L117-L157) вливает подключённые фрагменты в
   отправляемую строку и выдаёт типизированную обёртку;
   [`fragmentCode`](codegen/molplugin.js#L241-L263) выдаёт тип фрагмента и `unmask`;
   [`escapeDollars`](codegen/molplugin.js#L97-L99) - фикс с экранированием `$`.
   [`codegen/preset.js`](codegen/preset.js) настраивает один выходной файл на каждый входной.
3. Сгенерированный результат: [`app/notes.graphql.ts`](app/notes.graphql.ts#L13-L30)
   (замаскированный запрос с влитым фрагментом) и
   [`note/card/note.graphql.ts`](note/card/note.graphql.ts#L4-L13) (тип фрагмента + `unmask`).
4. Рантайм: [`graphql/index.ts`](graphql/index.ts):
   [слой запросов и соглашение о перезапросе](graphql/index.ts#L80-L135),
   [маркеры инвалидации](graphql/index.ts#L34-L68),
   [непрозрачный тип ссылки](graphql/index.ts#L136-L148).
5. Компонент, потребляющий фрагмент: [`note/card/card.view.ts`](note/card/card.view.ts):
   [`note()` снимает маску со ссылки](note/card/card.view.ts#L15-L17);
   [`renders()`](note/card/card.view.ts#L56-L69) - счётчик, тикающий на каждом перезапросе.
6. Опт-аут в действии: [`app/app.view.ts`](app/app.view.ts#L44-L60):
   `viewer_static()` передаёт `{ revalidate: false }`, поэтому его счётчик не двигается.

## Структура проекта

Репозиторий - **mam-пакет** в каноничной форме [hyoo-ru](https://github.com/hyoo-ru):
без вендоренного бутстрапа сборщика, только файлы модулей. Воркспейс mam
(клонированный из [hyoo-ru/mam](https://github.com/hyoo-ru/mam), который даёт
`mam.ts`/`tsconfig.json` и карту namespace→репозиторий для `mol`/`node`) монтирует
этот репозиторий в `demo/`. Именно это делает
[hyoo-ru/mam_build](https://github.com/hyoo-ru/mam_build) в CI с `package: 'demo'`.

```
codegen/
  graphqlgen.js         graphql-codegen config (checked-in SDL, no introspection)
  preset.js             one output per .graphql file + shared schema types
  molplugin.js          typed wrappers + fragment unmask helpers, in namespace $
server/
  schema.graphql        the SDL: single source of truth for server AND codegen
  index.mjs             graphql-yoga mock server with in-memory data
graphql/index.ts        runtime: request fn, error, reactive marker, ref type
graphql/schema.graphql.ts   (generated) shared scalar/enum/input types
app/                    $demo_app: page, plain query + fragment-composing query
app/static/             $demo_app_static: static entry, app + in-browser mock transport
note/card/              $demo_note_card: fragment + unmask + typed mutation
pages/                  FALLBACK static-site assembly (esbuild + graphql-js executor)
package.json            DEV TOOL only: codegen + mock server (not part of the build)
```

## Команды

Локальная разработка идёт внутри воркспейса mam:

```sh
git clone https://github.com/hyoo-ru/mam.git mam-ws && cd mam-ws
git clone https://github.com/trip2g/mol_graphql.git demo
npm install
```

| команда (из корня воркспейса) | что делает |
|---|---|
| `npm start` | dev-сервер mam на :9080 (`/demo/app/`); рядом запустите мок-сервер |
| `npx mam demo/app` | разовая продакшен-сборка в `demo/app/-/` (проверяет типы бандла, запускает тесты) |
| `npx mam demo/app/static` | сборка статической бессерверной входной точки в `demo/app/static/-/` |

| команда (из `demo/`, этот репозиторий) | что делает |
|---|---|
| `docker-compose up --build` | мок GraphQL API на :4000 + собранное приложение на :8080 |
| `npm install && npm run codegen` | перегенерировать все `*.graphql.ts` |
| `npm run codegen:watch` | перегенерация при каждом изменении .graphql/схемы |
| `npm run server` | запустить мок GraphQL-сервер локально |

Цикл разработки - `npm run codegen:watch` + `npm run server` (оба в `demo/`) +
`npm start` (корень воркспейса) в трёх терминалах. Сборщик mam подхватывает
перегенерированные `.graphql.ts` как любое изменение исходников.

## Сборка и деплой (GitHub Pages)

[`deploy.yml`](.github/workflows/deploy.yml) - каноничный пайплайн hyoo-ru:
`hyoo-ru/mam_build@master2` собирает воркспейс (клонирует `hyoo-ru/mam` +
зависимости, монтирует этот репозиторий как `package: 'demo'`), билдит `demo/app`
и `demo/app/static`, запускает все `*.test.ts`; затем папка `demo/app/static/-`
публикуется на Pages. Стандартного `GITHUB_TOKEN` достаточно: всё, что клонирует
mam_build, публично. Задеплоенный сайт - это `$demo_app_static`
([`app/static/static.ts`](app/static/static.ts)): один бандл, где транспортный шов
заменён на синхронный браузерный мок, отвечающий на каждую операцию из того же
набора данных, что и мок-сервер. Синхронизируйте его с
[`server/mock.mjs`](server/mock.mjs) вручную.

[`pages.yml`](.github/workflows/pages.yml) - ручной ЗАПАСНОЙ (доканоничный) путь:
mam-сборка `demo/app` + [`pages/build.mjs`](pages/build.mjs), который через esbuild
собирает настоящий graphql-js-исполнитель поверх SDL ([`pages/mock.mjs`](pages/mock.mjs)).
Удалите его (вместе с `pages/`), как только каноничный деплой подтверждён вживую.

## Как перенести это в свой проект

1. Возьмите `codegen/` как есть; направьте `schema` на свой SDL-файл, а
   `documents` - на своё UI-дерево; задайте в `molRuntime` свой префикс
   (например `$myapp_graphql`).
2. Портируйте `graphql/index.ts` под свой префикс (функция запроса, ошибка, тип
   ссылки, маркер инвалидации) или замените тело `*_request` на свой транспорт.
3. Пишите файлы `.graphql` рядом со своими компонентами: **одна операция или
   фрагмент на файл**; путь файла задаёт сгенерированный символ
   (`a/b/c.graphql` → `$a_b_c`) и имя операции (`a_b_c`, что бы вы ни написали
   в файле, хоть ничего); фрагменты тоже называйте по пути файла (то же правило
   `a_b_c`) - это автоматически даёт глобальную уникальность; при отклонении
   codegen предупреждает.
4. `npm run codegen:watch` рядом с dev-сервером mam. Сгенерированные `*.graphql.ts`
   можно коммитить (этот репозиторий так и делает), чтобы приложение собиралось
   без предварительного запуска codegen.

### Грабли (проверено на себе)

- **Пути модулей $mol буквальны**: `$demo_note_card` обязан лежать в
  `demo/note/card/` воркспейса, то есть в `note/card/` этого репозитория.
  Подчёркивание = разделитель директорий, всегда. (Сборщик разрешает FQN
  зависимостей по точным сегментам пути; поэтому и точка монтирования `demo`,
  а не, скажем, `mol_graphql`: в сегментах пути не должно быть подчёркиваний.)
- **Сканер зависимостей $mol читает `$`-токены везде**, включая строковые литералы
  и док-комментарии. Переменные GraphQL (`$id`) и ключи маскировки фрагментов
  (`' $fragmentRefs'`) стали бы фантомными зависимостями модулей и валили бы
  сборку. Поэтому codegen экранирует каждый `$` в выдаваемых GraphQL-строках и в
  выводе стокового плагина типов как `\u0024` (для TS/JS это идентично и на уровне
  типов, и в рантайме). Если пишете такие токены руками в модульном `.ts` -
  экранируйте так же (см. `graphql/index.ts`).
  Раньше это обходили вручную: добавляли пустые директории-заглушки с именами
  фантомных токенов (`fragment/`, `id/`, ...), чтобы сканер разрешал их в ничто.
  Экранирование каждого `$` - каноничный фикс: без директорий-заглушек, и новые
  имена полей и переменных он переживает автоматически.
- **`mam.ts`/`mam.jam.js` должны существовать в корне воркспейса** (они объявляют
  `class $`); без них любое использование `$` как типа в mol не компилируется.
  В каноничной раскладке они приходят из центрального воркспейса
  [hyoo-ru/mam](https://github.com/hyoo-ru/mam); пакетный репозиторий вроде этого
  НЕ должен вендорить свои.
- Сгенерированная обёртка операции, подключающей фрагменты, несёт док-комментарий
  `/** Spreads fragments: $demo_note_card_note */`; для сборщика это настоящее
  ребро зависимости (фрагменты независимы от иерархии вьюх, так что больше ничто
  не привязало бы модуль фрагмента к бандлу).
- Приложение ходит на `http://localhost:4000/graphql` (см. `$demo_graphql_endpoint`,
  переопределите для других конфигураций). CORS на мок-сервере открыт.

## Тестирование компонентов с запросами и мутациями

Тесты $mol живут рядом с кодом: [`card.view.test.ts`](note/card/card.view.test.ts),
[`app.view.test.ts`](app/app.view.test.ts). Они используют `$mol_test` + `$mol_assert`,
и каждая сборка mam запускает все `*.test.ts` (компилирует их в
`demo/app/-/node.test.js` и запускает; упавший тест валит сборку). Запустить только
бандл: `node --enable-source-maps demo/app/-/node.test.js` из корня воркспейса.

GraphQL-компоненты тестируются БЕЗ сервера - мокается транспортный шов.
`$demo_graphql_transport` - это `export let` в неймспейсе, поэтому тест
переприсваивает его на мок, который считает вызовы по операциям и отвечает из
крошечного in-memory-хранилища, а затем восстанавливает его в `finally`:

```ts
// app/app.view.test.ts (sketch)
const { calls, transport } = graphql_mock()          // per-operation call counter + in-memory store
with_transport(transport, () => {
	const app = $demo_app.make({ $ })
	$mol_assert_equal(app.greeting(), 'Reading list of Ada Lovelace')
	$mol_assert_equal(calls.demo_app_notes, 1)
	// mutate, then re-read, then assert the page queries refetched
})
```

Что доказывают тесты:
- **Снятие маски с фрагмента** (`card.view.test.ts`): ссылка на фрагмент
  разворачивается в типизированные поля, и карточка их рендерит. Без сети.
- **Соглашение о перезапросе** (`app.view.test.ts`): после мутации лайка запросы
  страницы уходят повторно (счётчики вызовов `demo_app_notes` и `demo_app_viewer`
  растут 1 → 2), и данные меняются.
- **Опт-аут** (`app.view.test.ts`): запрос с `{ revalidate: false }` выполняется
  ровно один раз, несмотря на мутацию, а мутация с `{ revalidate: false }`
  не трогает запросы страницы.

Полезно знать:
- Мок-транспорт синхронный (тот же контракт, что у мока для Pages), поэтому кейсы -
  обычные синхронные функции. Чтение `$mol_mem` в тесте просто вычисляет его.
- Перезапрос ленив: мутация помечает ячейки запросов страницы устаревшими, но
  ничего не пересчитывается до следующего чтения, поэтому проверяйте счётчики
  запросов ПОСЛЕ повторного чтения.
- `$mol_mem` пропускает пересчёты с глубоко равным результатом, поэтому для
  наблюдаемого перерендера нужен мок с состоянием (лайки реально увеличиваются).
- Изолированный на каждый кейс `$` не может перехватить транспорт (свободные
  функции читают глобальный `$`), поэтому мок - глобальная подмена,
  восстанавливаемая в `finally`. Утечек нет, потому что $mol выполняет кейсы
  последовательно.

## Что замокано и упрощено

- GraphQL-сервер - graphql-yoga с фиксированными данными в памяти (лайки реально
  увеличиваются на стороне сервера).
- Ни подписок, ни persisted queries, ни нормализованного кэша (см. выше).
- `type-check evidence`: сама сборка mam проверяет типы ровно той программы, что
  попадает в бандл (её аудит валит сборку на любой ошибке TS). `tsc -p .` по всему
  воркспейсу затягивает ещё и несобранные демо-модули mol, так что там будет шум;
  настоящие ворота - аудит бандла.

## Что почитать

Модель фрагментов здесь следует Relay, но без его рантайм-стора. Философия:
компоненты владеют своими требованиями к данным, объявленными рядом с кодом,
который их использует.

- [Relay: Thinking in Relay](https://relay.dev/docs/principles-and-architecture/thinking-in-relay/).
  Почему каждый компонент объявляет свои данные фрагментом и как фрагменты компонуются.
- [Relay: Guided tour, rendering fragments](https://relay.dev/docs/guided-tour/rendering/fragments/).
  Та самая модель рендеринга и маскировки, которую воспроизводит это демо.
- [Relay: Thinking in GraphQL](https://relay.dev/docs/principles-and-architecture/thinking-in-graphql/).
  Философия загрузки данных, стоящая за всем этим.
- [urql: Document caching](https://commerce.nearform.com/open-source/urql/docs/basics/document-caching/).
  Кэш легче нормализованного стора Relay - ближе к здешнему выбору «просто перезапроси».
- [Павел Черторогов: ApolloClient или Relay с фрагментами (YouTube, RU)](https://www.youtube.com/watch?v=VdoPraj0QqU).
  Фрагменты, «волосатый» GraphQL и TypeScript.
